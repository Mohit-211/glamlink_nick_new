## Improvement 1: Storybook Stories for Components

**Priority:** P2
**Effort:** Large (ongoing)
**Category:** Developer Experience
**Impact:** High

### Problem

Testing component appearance and behavior requires navigating through the full application to reach specific states. Want to see what a failed message looks like? You need to disconnect from the network, send a message, and wait. Want to see an empty state? You need to delete all data. This slows development and makes it easy to miss edge cases.

### Solution

Storybook provides an isolated component development environment where each component can be viewed in all its states—loading, error, empty, populated, focused, selected, etc.—with interactive controls to modify props. This speeds up development, makes it easy to verify edge cases, enables visual regression testing, and serves as living documentation for the component library.

### General Implementation Approach

#### 1. Setup Storybook (One-time)

```bash
npx storybook@latest init
```

Configure for Next.js in `.storybook/main.ts`:
```typescript
import type { StorybookConfig } from '@storybook/nextjs';

const config: StorybookConfig = {
  stories: [
    '../lib/**/*.stories.@(js|jsx|ts|tsx)',
    '../app/**/*.stories.@(js|jsx|ts|tsx)',
  ],
  addons: [
    '@storybook/addon-essentials',
    '@storybook/addon-interactions',
    '@storybook/addon-a11y', // Accessibility checks
  ],
  framework: {
    name: '@storybook/nextjs',
    options: {},
  },
  docs: {
    autodocs: 'tag',
  },
};

export default config;
```

#### 2. Story File Convention

Create stories alongside components:
```
lib/features/[feature]/components/
├── MyComponent.tsx
├── MyComponent.stories.tsx    # Stories file
└── ...
```

#### 3. Story Template

Use this template for new stories:
```typescript
import type { Meta, StoryObj } from '@storybook/react';
import { MyComponent } from './MyComponent';

const meta: Meta<typeof MyComponent> = {
  title: '[Feature Name]/MyComponent',
  component: MyComponent,
  parameters: {
    layout: 'padded', // or 'centered', 'fullscreen'
  },
  tags: ['autodocs'],
  argTypes: {
    // Define controls for props
    variant: {
      control: 'select',
      options: ['primary', 'secondary', 'danger'],
    },
    disabled: { control: 'boolean' },
  },
};

export default meta;
type Story = StoryObj<typeof MyComponent>;

// Default state
export const Default: Story = {
  args: {
    // Default props
  },
};

// Loading state
export const Loading: Story = {
  args: {
    isLoading: true,
  },
};

// Error state
export const Error: Story = {
  args: {
    error: 'Something went wrong',
  },
};

// Empty state
export const Empty: Story = {
  args: {
    items: [],
  },
};

// With data
export const WithData: Story = {
  args: {
    items: [/* mock data */],
  },
};
```

#### 4. Priority Order for Adding Stories

1. **Shared UI components** (`lib/components/ui/`)
2. **Form components** (inputs, selects, date pickers)
3. **Feature-specific components** (starting with most reused)
4. **Page-level components** (for integration testing)

#### 5. Recommended Stories per Component

| Component Type | Recommended Stories |
|----------------|---------------------|
| Button | Default, Disabled, Loading, Each variant |
| Input | Default, Focused, Error, Disabled, With value |
| Card/List Item | Default, Selected, Highlighted, With actions |
| Modal/Dialog | Open, With form, With error |
| Data Display | Loading, Empty, With data, Error |
| Form | Empty, Filled, Validating, Submitted |

### Directories to Cover

| Directory | Priority | Components |
|-----------|----------|------------|
| `lib/components/ui/` | High | ~15 shared components |
| `lib/admin/components/fields/` | High | Form field components |
| `lib/features/crm/profile/support-messaging/components/` | Medium | Messaging UI |
| `lib/pages/brand/components/` | Medium | Brand page components |
| `lib/admin/components/` | Lower | Admin panel tabs |

## Improvement 2: Unit Tests for Hooks & Utilities

**Priority:** P2
**Effort:** Large (ongoing)
**Category:** Developer Experience
**Impact:** High

### Problem

Custom hooks and utility functions contain critical business logic—rate limiting, validation, formatting, state management—but have no automated tests. Changes could introduce subtle bugs that break core functionality without any warning until users report issues in production.

### Solution

Comprehensive unit tests verify behavior in isolation: that calculations are correct, that edge cases are handled, that invalid inputs are rejected. Tests catch regressions immediately when code changes, provide documentation of expected behavior, and give developers confidence to refactor.

### General Implementation Approach

#### 1. Setup Testing Framework (One-time)

Install dependencies:
```bash
npm install -D vitest @testing-library/react @testing-library/react-hooks jsdom
```

Create `vitest.config.ts`:
```typescript
import { defineConfig } from 'vitest/config';
import react from '@vitejs/plugin-react';
import path from 'path';

export default defineConfig({
  plugins: [react()],
  test: {
    environment: 'jsdom',
    globals: true,
    setupFiles: ['./vitest.setup.ts'],
    include: ['**/*.test.{ts,tsx}', '**/__tests__/**/*.{ts,tsx}'],
    coverage: {
      provider: 'v8',
      reporter: ['text', 'html'],
      exclude: ['**/*.stories.tsx', '**/*.test.tsx'],
    },
  },
  resolve: {
    alias: {
      '@': path.resolve(__dirname, './'),
    },
  },
});
```

Create `vitest.setup.ts`:
```typescript
import '@testing-library/jest-dom';
```

Add scripts to `package.json`:
```json
{
  "scripts": {
    "test": "vitest",
    "test:coverage": "vitest run --coverage",
    "test:ui": "vitest --ui"
  }
}
```

#### 2. Test File Convention

Place tests alongside source files:
```
lib/features/[feature]/hooks/
├── useMyHook.ts
├── useMyHook.test.ts    # Test file
└── ...

lib/features/[feature]/utils/
├── myUtility.ts
├── myUtility.test.ts    # Test file
└── ...
```

Or use `__tests__` directories:
```
lib/features/[feature]/hooks/
├── __tests__/
│   └── useMyHook.test.ts
├── useMyHook.ts
└── ...
```

#### 3. Hook Test Template

```typescript
import { describe, it, expect, vi, beforeEach, afterEach } from 'vitest';
import { renderHook, act } from '@testing-library/react';
import { useMyHook } from '../useMyHook';

describe('useMyHook', () => {
  beforeEach(() => {
    vi.useFakeTimers();
  });

  afterEach(() => {
    vi.useRealTimers();
    vi.clearAllMocks();
  });

  it('should initialize with default values', () => {
    const { result } = renderHook(() => useMyHook());

    expect(result.current.value).toBe(initialValue);
    expect(result.current.isLoading).toBe(false);
  });

  it('should update value on action', () => {
    const { result } = renderHook(() => useMyHook());

    act(() => {
      result.current.setValue('new value');
    });

    expect(result.current.value).toBe('new value');
  });

  it('should handle async operations', async () => {
    const { result } = renderHook(() => useMyHook());

    await act(async () => {
      await result.current.fetchData();
    });

    expect(result.current.data).toBeDefined();
  });

  it('should cleanup on unmount', () => {
    const { unmount } = renderHook(() => useMyHook());

    unmount();

    // Verify cleanup happened
  });
});
```

#### 4. Utility Test Template

```typescript
import { describe, it, expect } from 'vitest';
import { myUtility, formatValue, validateInput } from '../myUtility';

describe('myUtility', () => {
  describe('formatValue', () => {
    it('should format normal input correctly', () => {
      expect(formatValue('test')).toBe('TEST');
    });

    it('should handle empty input', () => {
      expect(formatValue('')).toBe('');
    });

    it('should handle null/undefined', () => {
      expect(formatValue(null)).toBe('');
      expect(formatValue(undefined)).toBe('');
    });
  });

  describe('validateInput', () => {
    it('should return true for valid input', () => {
      expect(validateInput('valid')).toBe(true);
    });

    it('should return false for invalid input', () => {
      expect(validateInput('')).toBe(false);
      expect(validateInput('a'.repeat(1000))).toBe(false);
    });
  });
});
```

#### 5. Priority Order for Adding Tests

1. **Utility functions** (pure functions, easy to test)
2. **Validation/sanitization** (security-critical)
3. **Core hooks** (business logic)
4. **State management** (Redux slices)

#### 6. What to Test

| Type | Test For |
|------|----------|
| Utilities | Input/output for all cases, edge cases, error handling |
| Hooks | Initial state, state transitions, side effects, cleanup |
| Validation | Valid inputs, invalid inputs, boundary conditions |
| Formatters | All format variations, locales, edge cases |

| Directory | Priority | Files |
|-----------|----------|-------|
| `lib/features/*/utils/` | High | All utility files |
| `lib/features/*/hooks/` | High | Core business logic hooks |
| `lib/services/` | High | Service layer functions |
| `lib/admin/utils/` | Medium | Admin utilities |

## Improvement 3: E2E Tests for Critical Flows

**Priority:** P3
**Effort:** Large
**Category:** Developer Experience
**Impact:** High

### Problem

Unit tests verify individual pieces work correctly, but don't test that the entire system works together. A user might be unable to complete a critical flow despite all individual components passing their tests, because the integration between them is broken.

### Solution

End-to-end tests using Playwright simulate real user interactions in an actual browser. They verify complete flows work from the user's perspective. Running these tests on every pull request catches integration issues before they reach production.

### General Implementation Approach

#### 1. Setup Playwright (One-time)

```bash
npm init playwright@latest
```

Configure `playwright.config.ts`:
```typescript
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: './e2e',
  fullyParallel: true,
  forbidOnly: !!process.env.CI,
  retries: process.env.CI ? 2 : 0,
  workers: process.env.CI ? 1 : undefined,
  reporter: 'html',
  use: {
    baseURL: 'http://localhost:3000',
    trace: 'on-first-retry',
    screenshot: 'only-on-failure',
  },
  projects: [
    {
      name: 'chromium',
      use: { ...devices['Desktop Chrome'] },
    },
    {
      name: 'Mobile Safari',
      use: { ...devices['iPhone 12'] },
    },
  ],
  webServer: {
    command: 'npm run dev',
    url: 'http://localhost:3000',
    reuseExistingServer: !process.env.CI,
  },
});
```

#### 2. Test File Structure

```
e2e/
├── fixtures/
│   ├── auth.ts           # Authentication helpers
│   └── test-data.ts      # Mock data generators
├── auth/
│   ├── login.spec.ts
│   └── signup.spec.ts
├── support-messaging/
│   ├── create-conversation.spec.ts
│   └── send-message.spec.ts
├── admin/
│   ├── brand-management.spec.ts
│   └── content-generation.spec.ts
└── global-setup.ts       # Setup before all tests
```

#### 3. Auth Fixture for Reuse

```typescript
// e2e/fixtures/auth.ts
import { test as base, Page } from '@playwright/test';

type AuthFixtures = {
  authenticatedPage: Page;
  adminPage: Page;
};

export const test = base.extend<AuthFixtures>({
  authenticatedPage: async ({ page }, use) => {
    // Login as regular user
    await page.goto('/login');
    await page.fill('[name="email"]', process.env.TEST_USER_EMAIL!);
    await page.fill('[name="password"]', process.env.TEST_USER_PASSWORD!);
    await page.click('button[type="submit"]');
    await page.waitForURL('/profile');
    await use(page);
  },

  adminPage: async ({ page }, use) => {
    // Login as admin
    await page.goto('/login');
    await page.fill('[name="email"]', process.env.TEST_ADMIN_EMAIL!);
    await page.fill('[name="password"]', process.env.TEST_ADMIN_PASSWORD!);
    await page.click('button[type="submit"]');
    await page.waitForURL('/admin');
    await use(page);
  },
});

export { expect } from '@playwright/test';
```

#### 4. E2E Test Template

```typescript
// e2e/[feature]/[flow].spec.ts
import { test, expect } from '../fixtures/auth';

test.describe('Feature: User Flow', () => {
  test('should complete the full flow', async ({ authenticatedPage: page }) => {
    // Navigate
    await page.goto('/feature');

    // Interact
    await page.click('button:has-text("Start")');
    await page.fill('[name="field"]', 'value');
    await page.click('button:has-text("Submit")');

    // Assert
    await expect(page.locator('.success-message')).toBeVisible();
    await expect(page).toHaveURL(/\/feature\/success/);
  });

  test('should handle errors gracefully', async ({ authenticatedPage: page }) => {
    await page.goto('/feature');

    // Trigger error condition
    await page.route('**/api/feature', route =>
      route.fulfill({ status: 500, body: 'Server error' })
    );

    await page.click('button:has-text("Submit")');

    await expect(page.locator('[role="alert"]')).toContainText('error');
  });
});
```

## Improvement 4: Bundle Splitting for Lazy Loading

**Priority:** P3
**Effort:** Medium
**Category:** Performance
**Impact:** Medium

### Problem

All feature code is included in the main JavaScript bundle and downloaded on initial page load, even if users never visit those features. This increases initial load time for all users to support features they may not use.

### Solution

Dynamic imports and code splitting load feature code only when users navigate to it. This reduces initial bundle size, speeds up first page load, and only loads code for features users actually access.

### General Implementation Approach

#### 1. Identify Candidates for Splitting

Good candidates for lazy loading:
- Features behind authentication
- Features accessed by minority of users
- Large feature modules (>50KB)
- Admin-only features
- Modals and dialogs

#### 2. Create Lazy Wrapper Components

```typescript
// lib/features/[feature]/components/lazy.ts
import dynamic from 'next/dynamic';
import { LoadingSpinner } from '@/lib/components/ui';

export const LazyFeaturePage = dynamic(
  () => import('./FeaturePage').then((mod) => mod.FeaturePage),
  {
    loading: () => <LoadingSpinner message="Loading..." />,
    ssr: false, // Disable SSR for client-only features
  }
);

export const LazyFeatureModal = dynamic(
  () => import('./FeatureModal').then((mod) => mod.FeatureModal),
  {
    loading: () => null, // No loading state for modals
    ssr: false,
  }
);
```

#### 3. Use in Route Pages

```typescript
// app/[feature]/page.tsx
import { LazyFeaturePage } from '@/lib/features/[feature]/components/lazy';

export default function FeaturePage() {
  return <LazyFeaturePage />;
}
```

#### 4. Prefetch on Hover

```typescript
// In navigation component
const prefetchFeature = () => {
  import('@/lib/features/[feature]/components/FeaturePage');
};

<Link
  href="/feature"
  onMouseEnter={prefetchFeature}
>
  Feature
</Link>
```

#### 5. Prefetch on Route Change

```typescript
// In layout or navigation
import { useRouter } from 'next/navigation';
import { useEffect } from 'react';

export function NavigationPrefetch() {
  const router = useRouter();

  useEffect(() => {
    // Prefetch likely next pages
    router.prefetch('/profile');
    router.prefetch('/support');
  }, [router]);

  return null;
}
```

| Feature | Path | Est. Size | Priority |
|---------|------|-----------|----------|
| Support Messaging | `/profile/support` | ~80KB | High |
| Admin Panel | `/admin/*` | ~150KB | High |
| Magazine | `/magazine/*` | ~100KB | Medium |
| Brand Editor | `/profile/brand/*` | ~60KB | Medium |
| AI Tools | `/admin/brainstorm` | ~40KB | Medium |

#### 7. Verify Splitting

Use bundle analyzer:
```bash
npm install -D @next/bundle-analyzer

# In next.config.js
const withBundleAnalyzer = require('@next/bundle-analyzer')({
  enabled: process.env.ANALYZE === 'true',
});

module.exports = withBundleAnalyzer(nextConfig);

# Run analysis
ANALYZE=true npm run build
```

Check for:
- Feature code in separate chunks
- Main bundle size reduced
- No duplicate code across chunks

## Implementation Priority

1. **Unit Tests** - Start here, highest ROI for stability

## Notes

These improvements are infrastructure investments that pay dividends over time. They should be implemented incrementally alongside feature work rather than as dedicated projects. Each PR can include:
- Storybook stories for new/modified components
- Unit tests for new hooks/utilities
- Lazy loading for new features

This "test as you go" approach builds coverage gradually without blocking feature development.
